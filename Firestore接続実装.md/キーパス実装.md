目的:
ヒロメルに、最小構成のキーパス機能を実装したい。
共通キー1つを使い、Googleログイン済みユーザーのうち、正しいキーパスを入力した人だけ投稿できるようにしたい。

最重要条件:
今回の目的は「キーパス追加」だけです。
Auth復旧したばかりなので、Googleログイン周りを壊さないことを最優先にしてください。

絶対に守ること:
- 既存の Googleログイン機能を壊さない
- 既存の AuthProvider / AuthButtons / firebase.ts を不要に触らない
- layout.tsx を不要に触らない
- HomeButton を触らない
- 画像投稿機能には触らない
- UI大改修をしない
- 新しい複雑な設計を作らない

前提:
- Next.js App Router
- Firebase Hosting
- Firestore 利用中
- Googleログインは現在動作している
- ログインすると右上にユーザー名とログアウトが出る
- /post では現在ログイン済みユーザーが投稿フォームを使える
- 今回やりたいのは、そこに「キーパス通過済み」条件を追加すること

要件:
- 共通キー1つでOK
- ログイン状態と組み合わせて制御
- 入力済みユーザーのみ投稿可能
- 個別キー不要
- 高度な権限管理不要

実装したい挙動:
1. 未ログインで /post に来たら、今まで通りログイン要求画面を表示する
2. ログイン済みだがキーパス未通過なら、投稿フォームではなくキーパス入力画面を表示する
3. 正しいキーパスを入力したら、そのユーザーを Firestore に記録する
4. 記録済みユーザーは次回以降キーパス不要で /post の投稿フォームを表示する
5. キーが違う場合はエラーメッセージを出す
6. 既存の投稿処理は壊さない

推奨する最小構成:
- Firestore コレクション名: creatorAccess
- ドキュメントID: uid
- 保存項目:
  - uid
  - email
  - displayName
  - approved: true
  - approvedAt: serverTimestamp()
  - approvedBy: "common_key"

キーパスの持ち方:
- まずは環境変数でよい
- 例: NEXT_PUBLIC_CREATOR_KEYPASS
- もし公開環境でクライアント露出が気になる場合は、その点を最後に短く報告する
- ただし今回は最小実装優先で止まらないこと

/post の表示分岐:
- 未ログイン -> ログイン要求画面
- ログイン済み + 未承認 -> キーパス入力画面
- ログイン済み + 承認済み -> 投稿フォーム

UI要件:
- 既存の /post ページの見た目を大きく壊さない
- キーパス入力画面も既存トーンに合わせてシンプルにする
- 文言は日本語でよい
- 例:
  - 見出し: キーパスの入力
  - 説明: 投稿するには運営から共有されたキーパスが必要です
  - ボタン: キーパスを確認する

Firestore rules:
- creatorAccess の read/write が必要なら最小限追加する
- ads の既存ルールは壊さない
- clicks の既存ルールは壊さない
- rules変更があるなら最後に明記する

実装の進め方:
1. まず既存Authや/postの現状コードを読む
2. 変更範囲を /post と必要最小限の Firestore / rules に絞る
3. Auth関連ファイルは、必要でない限り変更しない
4. 最小変更で完成させる
5. build を通す
6. 必要なら hosting / rules のデプロイ手順を書く

完了条件:
- 未ログインでは /post でログイン要求
- ログイン済み未承認では /post でキーパス入力要求
- 正しいキーパス入力で Firestore に承認記録が保存される
- 承認済みユーザーは /post で投稿フォームが表示される
- 再アクセス時にキーパス再入力不要
- 既存のログイン機能が壊れていない
- 既存の投稿機能が壊れていない
- build が通る

- 修正ファイル一覧
- どこに何を追加したか
- rules変更の有無
- 必要なデプロイコマンド
- Akiが最後に確認すること3つ
